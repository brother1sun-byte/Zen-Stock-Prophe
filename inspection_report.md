## 2. 対象ファイル群の構成と責務の評価

### 2.1 UI コンポーネントとメインロジック
- **`src/App.jsx`**
  - **現状の役割**: アプリのメインレイアウト、タブ切り替え、カスタムフックの呼び出し、各コンポーネントへのプロパティの受け渡しなど、アプリケーション全体のハブ（中枢）として機能しています。
  - **評価**: UIと状態管理の連結が正しく行われています。ただし、ファイルサイズが巨大（3,000行以上）であり、UIヘルパー関数（`yen`, `pct`, `riskLevelLabel` 等）や、デモ用フォールバックデータ、`StatusPill` のような小さなコンポーネントがハードコードされています。

### 2.2 カスタムフックとデータ同期
- **`src/hooks/useSelectedStock.js`**
  - **役割**: 選択された銘柄（Ticker）の状態管理、監視リストやランキング等からの銘柄選択の同期を担当。
  - **評価**: 複数の出所（MarketItem, Detail, Watchlist）から選択された銘柄のデータを単一の `selectedStock` として正規化するロジックが見事に分離されています。
- **`src/hooks/usePracticeOrder.js`**
  - **役割**: 練習注文のフォーム状態、バリデーション、履歴リストの管理。
  - **評価**: 注文ステータス（DRAFT, PENDING, FILLED, CANCELLED等）が定数化されており、`validatePracticeOrder` 関数により「データ品質警告」や「未入力項目」のチェックが堅牢に実装されています。
- **`src/hooks/usePortfolioLedger.js`**
  - **役割**: ポートフォリオの保有状況、スコアリング、分散状況の評価、及びポジションの終了（SOLD, VOIDED, ARCHIVED）管理。
  - **評価**: 単純な状態管理だけでなく、ボラティリティや最大ドローダウンの計算ロジックが組み込まれており、検証ツールとして非常に実用的です。
- **`src/hooks/useMarketData.js`**
  - **役割**: バックエンドAPIからの非同期データ取得（株価、ランキング、ポートフォリオ等）とキャッシュへの書き込み。
  - **評価**: API呼び出し時の並列処理（`Promise.allSettled`）が効果的に使われており、一部エラー時でも他のデータを生かす設計（フォールバック）が優れています。

### 2.3 API通信とユーティリティ
- **`src/api/apiClient.js`**
  - **役割**: バックエンド通信のフェッチ処理、キャッシュの読み書き、複数エンドポイント（ローカル・プロキシ等）の自動解決。
  - **評価**: リトライ／フォールバック可能な複数ベースURLの試行ロジックが優秀です。
- **`src/utils/dataSource.js`**
  - **役割**: データソース（yfinance, Stooq, 補完データなど）のラベル付けと警告メッセージの生成。

### 2.4 バックエンドとインフラ

  - **役割**: FastAPIを用いたバックエンドサーバー。市場スクリーニング、分析ロジックとの統合。
  - **評価**: 本アプリが「シミュレーター専用」であることを徹底しており（`LIVE_BROKER_ORDERS_ENABLED = False`）、スコア計算やデータ品質の検証プロセスが詳細に実装されています。

## 3. 重要アーキテクチャの分析詳細

### App.jsx の整理・リファクタリング方針
- **残すべきもの**: レイアウトの基本骨格、タブ管理、カスタムフックによる状態の連携処理。
- **移動すべきもの**:
  - `yen`, `pct`, `cacheStatusLabel` などのフォーマット系ヘルパー関数 ➔ `src/utils/formatters.js` 等へ。
  - `daytradeFallback`, `demo`, `aiFundDeskFallback` などの静的データ ➔ `src/constants/fallbackData.js` 等へ。
  - `StatusPill`, `ProTooltip` などのUIコンポーネント ➔ `src/components/` ディレクトリへ。

### データ品質と警告UIのロジック
- 「補完データ（Synthetic）」「一時保存（Cache）」「出所不明（Unknown）」「J-Quants遅延データ」に関する警告ロジックは、`src/utils/dataSource.js` で集中的に判定されています。
- これが `usePracticeOrder` や `App.jsx` に伝播し、**「ユーザーが架空・古いデータで実注文の判断を下さないよう、強制的に警告テキストを表示する」** という安全機構（ガードレール）として完璧に機能しています。

### 練習注文とポートフォリオ台帳のロジック
- **練習注文 (Practice Order)**: UI入力 ➔ `usePracticeOrder` 内でのバリデーション（ここで前述の品質警告が加味される） ➔ ローカル（または疑似API）への状態保存（`PENDING` ➔ `FILLED`）、という独立したデータフローが確立されています。
- **ポートフォリオ台帳 (Portfolio Ledger)**: 疑似的に保持しているポジションのパフォーマンス（ドローダウンやリスクスコア）を計算する役割を持っており、単なるCRUDアプリにとどまらない、投資シミュレーターとしての価値を生み出しています。

### エラーハンドリングとUIの堅牢性
- **API通信障害時**: `useMarketData` の並列取得で `Promise.allSettled` を用いており、一部のAPIが死んでいてもアプリ全体はクラッシュしません。エラー時には `setStatus` で画面上部に「一部データ不足」として安全に通知します。
- **レンダリング障害時**: `App.jsx` に組み込まれた `<AppErrorBoundary>` により、万が一Reactツリー内で例外が発生しても白画面にならず、「画面を復旧できませんでした」というエラーレポートUIにフォールバックします。

## 4. 総評 (Conclusion)

全体として、**「ユーザーに誤った投資判断をさせないシミュレーター」** というプロダクトの目標が、UIからバックエンドのロジック、そしてテストコードに至るまで首尾一貫して実装されています。

現状で即座に修正すべきバグや不具合はありません（検証テストは全てパスしています）。今後のエンハンスメントとして、**`App.jsx` の肥大化を解消するリファクタリング**（ヘルパーや定数の切り出し）を行うことで、さらなる保守性の向上が期待できます。
